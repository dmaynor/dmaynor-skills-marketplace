# SAT Analysis Skill — Ground-Up Redesign Design Spec

**Date:** 2026-05-05
**Author:** David Maynor
**Status:** Approved design, pending implementation plan
**Plugin:** `plugins/sat-analysis/`
**Target version:** `2.0.0` (semantic break from `1.0.0`)

---

## 1 / Context

### 1.1 What triggered the redesign

A GPT-5 critique evaluating a "mobile SAT app strategy" surfaced doctrinal and structural defects that also apply to the current Claude Code skill at `plugins/sat-analysis/`. An audit of the skill confirmed those defects and found additional ones beyond what the critique flagged.

### 1.2 Audit findings (defects in current skill)

The skill is internally inconsistent. Its three primary artifacts disagree on what rigorous looks like:

| Artifact | Source quality | Sensitivity | Reliability scoring | Likelihood ≠ confidence |
|---|---|---|---|---|
| `SKILL.md` | ❌ | ❌ | ❌ | ❌ |
| `assets/templates/assessment_full.md` | ✅ | ✅ | ❌ | ❌ |
| `scripts/ach_matrix.py` | ✅ (A-F) | ✅ | ❌ (captured, not used) | ❌ |
| `references/techniques.md` | — | ✅ | — | ❌ |

Specific defects:

**Doctrinal:**
1. Likelihood and confidence are conflated in SKILL.md, the template, and `techniques.md`. ICD 203 §c.6 prohibits combining them in the same expression.
2. No source-quality field in SKILL.md (template has it; script captures A-F but ignores it in scoring).
3. No implications / indicators / customer-relevance output (ICD 203 customer-relevance requirement).
4. No coherence check on hypothesis probabilities (Mandel 2018 — strongest empirical SAT-improvement finding).

**ACH-specific:**
5. `ach_matrix.py:101` sums `+` and `-` symmetrically. This violates Heuer's disconfirmation rule and rewards vague hypotheses (a hypothesis that gets `+` against everything wins).
6. `Evidence.reliability` (NATO Admiralty A-F) is captured in the script and serialized to JSON but never used in scoring. Looks rigorous, isn't.
7. `get_diagnosticity()` and `sensitivity_analysis()` exist in the script but the SKILL.md never tells Claude to surface them.

**Structural:**
8. SKILL.md, template, script, and references diverge on rigor and required fields.
9. Scripts (`parse_logs.py`, `timeline.py`, `ach_matrix.py`) are never invoked by the SKILL.md workflow — dead weight relative to the skill, though useful as standalone library code.
10. No rigor dial — same workflow whether analyzing a kernel panic or a bar argument.
11. No anti-pattern / failure-mode warnings (false precision, box-checking, ACH theater).

**Parser-specific:**
12. `parse_logs.py:172-177` auto-tags URLs with `path_traversal_attempt`, `sqli_attempt`, `xss_attempt` based on substring matching. This is interpretation, not observation — directly violates the skill's own O/I separation rule (`techniques.md:151-156`).
13. `parsers.py:85` defines a `windows_event` regex that no parser function uses. Dead code.
14. `LogEntry.message` ↔ `TimelineEvent.description` and `LogEntry.user` ↔ `TimelineEvent.actor` work via fallback magic in `from_observations_json`, not canonical schema alignment.

### 1.3 Why redesign rather than incremental fix

The defects are interlocking. Fixing the likelihood/confidence conflation in SKILL.md without also fixing it in the template and references leaves the artifacts inconsistent. Fixing the script's Heuer bug without changing how the SKILL.md instructs Claude leaves the math right but the behavior unchanged. The core problem is that SKILL.md, scripts, template, and references were authored independently and drift apart over time.

Additionally, the user has signaled an eventual mobile app that will reuse this skill's logic. Incremental patching of a skill that grew organically does not produce a clean cross-platform contract.

## 2 / Decisions log (Q1–Q7)

| # | Question | Decision |
|---|---|---|
| Q1 | Rigor dial shape | **(a) Two modes: LIGHT and FULL.** Auto-default LIGHT when no artifact attached, FULL when artifact present. Override via `/sat light` or `/sat full`. |
| Q2 | Script-first vs inline math | **(a) Script-first for FULL mode.** LIGHT has no matrix. Avoids hallucinated arithmetic and "false precision" failure mode. |
| Q3 | Output schema | **(a) Three-layer output for FULL.** Decision card, analytic trace, tasking view — congruent across all three. |
| Q4 | Modes within FULL | **(a) Keep all 5 technical-domain submodes** (BREACH, CRASH, FIX, STATEMENT, GENERAL). LIGHT runs orthogonal to submodes. |
| Q5 | Doctrinal anchoring | **(a′) Anchor-by-ID.** Skill emits stable rule IDs (`[ICD-203-LC]`, `[HEUER-DISCONF]`, etc.). Citation enrichment runs post-facto via background process or sync, not at write time. |
| Q6 | Scripts fate | **(a) Treat scripts as a first-class library.** They are the shared engine for the desktop skill and the eventual mobile app. |
| Q7 | Scope of redesign | **(b) Skill + SDK contract.** Versioned JSON Schemas, importable Python module, mobile reads schemas natively. |
| Approach | Implementation strategy | **Approach 4 — Hybrid.** Schemas as source of truth (Approach 3's discipline), Python as reference implementation (Approach 1's structure). Mobile reimplements natively against the same schemas + parity tests. |

## 3 / Architecture

The redesigned skill is layered with one rule: **schemas are the source of truth, every other artifact conforms to them.**

```
plugins/sat-analysis/skills/sat-analysis/
│
├── schemas/                ← LAYER 1: contracts (versioned JSON Schema)
│   └── *.v1.json
│
├── doctrine/               ← LAYER 2: data (citations as JSON, not Python)
│   └── catalog.v1.json
│
├── sat_engine/             ← LAYER 3: reference implementation (Python)
│   ├── models.py           ← validated against schemas in CI
│   ├── parsers.py
│   ├── timeline.py
│   ├── ach.py
│   ├── doctrine.py
│   ├── enrich.py
│   └── validators.py
│
├── tests/
│   └── parity/             ← LAYER 4: golden input→output (any reimpl must match)
│
├── SKILL.md                ← LAYER 5: thin orchestrator — tells Claude what to call
├── references/             ← guidance Claude loads on demand
├── assets/templates/       ← three-layer markdown output forms
└── bin/sat                 ← CLI wrapping sat_engine
```

### 3.1 Layer dependencies (strict, one direction)

- **Layer 1** (schemas) depends on nothing. Pure contracts.
- **Layer 2** (doctrine) conforms to a schema in Layer 1.
- **Layer 3** (Python) conforms to schemas, reads doctrine.
- **Layer 4** (parity tests) verifies Layer 3 produces schema-valid output for known inputs.
- **Layer 5** (SKILL.md) orchestrates Layer 3 calls. References and templates are Claude-facing; engine is Python-facing.

### 3.2 Why this layering matters

- Mobile app reads schemas, doctrine, and parity tests. It never touches Python.
- Schema bumps require explicit version bumps and migration notes — drift becomes loud, not silent.
- The current skill's defect (SKILL.md / template / script disagreeing) becomes structurally impossible: the schemas validate everything.
- LIGHT mode skips most layers — it uses only SKILL.md and a small subset of schemas. FULL mode uses all layers.

## 4 / Mode workflows

### 4.1 Selector logic

```
Input contains attached artifact (logs, dump, diff, JSON, file path)?  → FULL
User explicitly typed /sat full?                                       → FULL
User explicitly typed /sat light?                                      → LIGHT
Otherwise                                                              → LIGHT
```

Inside FULL, submode auto-selects from input shape:

| Input | Submode |
|---|---|
| Logs + security question | `FULL/BREACH` |
| Crash dump / stack trace | `FULL/CRASH` |
| Code diff + fix question | `FULL/FIX` |
| Claim or statement | `FULL/STATEMENT` |
| Ambiguous | `FULL/GENERAL` |

### 4.2 LIGHT workflow (bar argument, 1 round-trip)

1. **Frame** — what is the disputed claim?
2. **Generate ≥3 hypotheses** — must include one null and one uncomfortable.
3. **List evidence both sides have** — informally; sources are "you", "me", "vague memory of an article."
4. **For each hypothesis, name the falsifier** — "what would settle this for me?"
5. **Output** — brief markdown decision card with **likelihood only** (no confidence — there is no source quality to anchor confidence to).

No scripts run. No schemas required at write time, though the output conforms to a stripped-down `decision_card.v1.json` shape. Skill purpose: enforce the cheap moves (alternative hypothesis, falsifier) without rigor theater.

### 4.3 FULL workflow (serious work, multi-step pipeline)

```
Input
  ↓
[1] Submode detection                                  (SKILL.md branches)
  ↓
[2] Ingest                                             sat_engine.parsers
    └→ Observation[] (schema-validated)
  ↓
[3] Timeline (if events have timestamps)               sat_engine.timeline
    └→ Timeline + gaps + rapid-succession
  ↓
[4] O/I separation + reliability assignment            (analyst, guided)
    └→ Observation[] with A-F reliability tags
  ↓
[5] Hypothesis generation                              (analyst, guided)
    └→ Hypothesis[] (mode-specific category coverage required)
  ↓
[6] Coherence check                                    sat_engine.ach
    └→ probabilities sum to ~100% or fail-loud
  ↓
[7] ACH matrix construction + Heuer scoring            sat_engine.ach
    └→ ACHMatrix with disconfirmation count, reliability-weighted
  ↓
[8] Sensitivity + diagnosticity                        sat_engine.ach
    └→ robust? which evidence discriminates?
  ↓
[9] Three-layer output emission                        sat_engine + templates
    └→ decision_card.md, analytic_trace.md, tasking_view.md
    └→ congruence guard: likelihood/confidence/key judgments match across all 3
  ↓
[10] Rule-ID markers embedded in trace                 (Claude, per SKILL.md)
  ↓
[11] (optional, later) Citation enrichment             sat_engine.enrich
     └→ rule_ids replaced with full ICD/Heuer/NATO citations
```

### 4.4 Submode-specific differences inside FULL

The submode only changes step [5]'s required hypothesis categories:

| Submode | Required category coverage |
|---|---|
| BREACH | Malicious External, Malicious Internal, Non-Mal Authorized, Non-Mal Unauthorized, System Artifact, Null |
| CRASH | Memory, Threading, Resource, Logic, External, Not-a-bug |
| FIX | Complete, Partial, Symptom, Ineffective, Regression Risk, Bypass Possible |
| STATEMENT | Logical validity, Evidential support, Assumption robustness, Alternative plausibility |
| GENERAL | WHO / WHAT / WHY / HOW dimensions |

Reference docs for each (`attack_patterns.md`, `crash_patterns.md`, `fix_patterns.md`) load on demand.

### 4.5 Anti-patterns block (new in SKILL.md)

A short rule list called out in both modes to prevent the failures the audit flagged:

- "Every ACH cell is `+` or `++`" → hypotheses too vague, split or sharpen.
- "Confidence ≥ 80% with single source" → confidence cap violated.
- "Probabilities don't sum to ~100%" → coherence check failed, refuse output.
- "Decision card likelihood ≠ analytic trace likelihood" → congruence broken, refuse output.
- "Source reliability set but never used in scoring" → fail (script enforces).

## 5 / Schemas (data model)

Nine schemas under `schemas/`. Every emitted artifact carries `schema_version: "1"` so contract changes are explicit.

### 5.1 Core data schemas

**`observation.v1.json`** — pure data, no interpretation
```
{ schema_version, id, timestamp, source, text, reliability, tags }
```
- `reliability`: enum **A–F** (NATO Admiralty: A=completely reliable → F=cannot be judged).
- `tags`: **restricted** — descriptive only (`"ssh_auth_event"` ✅), no classification (`"sqli_attempt"` ❌). Fixes the `parse_logs.py` O/I-violation defect (defect #12 in §1.2).

**`hypothesis.v1.json`**
```
{ schema_version, id, description, category, probability, falsifier }
```
- `probability`: 0.0–1.0, **required** so the coherence check has something to validate.
- `category`: enum, values depend on submode.
- `falsifier`: required string — what would disprove this. Forces the check at hypothesis-creation time.

**`evidence.v1.json`** — link Observation → Hypothesis ratings
```
{ schema_version, observation_id, ratings: {hypothesis_id: rating}, reliability_inherited }
```
- `ratings`: `"++" | "+" | "N" | "-" | "--"`.
- `reliability_inherited`: A–F, copied from the upstream observation. Used by the weighted-score calculation.

**`timeline.v1.json`**
```
{ schema_version, events[], gaps[], rapid_succession[],
  gap_threshold_seconds, rapid_threshold_seconds }
```
- Thresholds are explicit fields, not hardcoded — kernel panic = microseconds, breach = hours.

### 5.2 Analysis container

**`ach_matrix.v1.json`**
```
{ schema_version, title, submode, hypotheses[], evidence[],
  scores: {hyp_id: {sum, disconfirm_count, weighted}},
  winner, diagnosticity, sensitivity, coherence_check }
```
- Three scores per hypothesis: `sum` (informational), `disconfirm_count` (Heuer's rule, **the actual ranking criterion**), `weighted` (reliability-weighted sum).
- `winner` = hypothesis with **lowest** `disconfirm_count`. Ties broken by `weighted`.
- `sensitivity`: `{ robust: bool, flips_on_remove: [evidence_id] }`.
- `coherence_check`: `{ passed: bool, sum_of_probs: float }` — must be in [0.95, 1.05] or output is refused.

### 5.3 Three-layer output schemas

**`decision_card.v1.json`** — one screen
```
{ schema_version, analysis_id, bottom_line,
  likelihood: {term, range},
  confidence: {level, reasoning},
  top_implication, next_indicator }
```
- `likelihood` and `confidence` are **separate fields** — fixes the central doctrinal bug. ICD 203 §c.6 enforced structurally, not by prose discipline.

**`analytic_trace.v1.json`** — audit trail
```
{ schema_version, analysis_id, mode, submode,
  observations[], hypotheses[], ach_matrix | null,
  assumptions[], falsification[], limitations[],
  rule_ids[], congruence_hash }
```
- `ach_matrix` is `null` in LIGHT mode.
- `rule_ids[]`: the `[ICD-203-LC]`, `[HEUER-DISCONF]` markers embedded in the trace.
- `congruence_hash`: hash of `(likelihood, confidence, top_implication)`. Decision card and tasking view must hash to the same value or the engine refuses to render.

**`tasking_view.v1.json`** — follow-up
```
{ schema_version, analysis_id, collection_gaps[],
  indicators_to_watch: [{indicator, threshold, changes}],
  peer_review_requests[],
  change_conditions: [{if_observed, judgment_becomes}] }
```

### 5.4 Doctrine schema

**`doctrine.v1.json`** — schema for the catalog
```
{ schema_version, rules: [
    { rule_id, standard, section, title, summary, citation, severity_if_violated }
]}
```
- `severity_if_violated`: `"BLOCKING" | "WARNING" | "NOTE"`.

### 5.5 Invariants enforced by schemas

1. **Likelihood and confidence cannot be conflated** — different fields, different value spaces. The bug is structurally impossible.
2. **Source reliability is captured AND used** — `weighted` reads `reliability_inherited`. The "captured but ignored" defect is impossible.
3. **Three-layer output cannot drift** — `congruence_hash` mechanical check.
4. **Sum-of-positives can't quietly become the winner** — `winner` is computed from `disconfirm_count`; the sum is informational only.
5. **Vague hypotheses lose** — they pile up zero disconfirmations and zero discriminating evidence; `diagnosticity` and `disconfirm_count` together expose them.

## 6 / Three-layer output forms

The JSON is canonical. Markdown templates render it. The user sees markdown; the engine and any future mobile app exchange JSON.

### 6.1 Decision card (`assets/templates/decision_card.md`)

One screen, headline-first. ~10 lines.

```markdown
# {{title}}

**Bottom line:** {{bottom_line}}

| Field | Value |
|---|---|
| Likelihood | {{likelihood.term}} ({{likelihood.range}}) |
| Confidence | {{confidence.level}} — {{confidence.reasoning}} |
| Top implication | {{top_implication}} |
| Next indicator | {{next_indicator}} |

— Analysis `{{analysis_id}}` · trace: `analytic_trace.md`
```

Likelihood and confidence appear in different rows of the same table, never combined. The decision card carries no `[RULE-ID]` markers — it is for the decision-maker, not the auditor.

### 6.2 Analytic trace (`assets/templates/analytic_trace.md`)

The audit trail. The canonical artifact. Where `[RULE-ID]` markers live.

```markdown
# Analytic Trace — {{analysis_id}}

**Mode:** {{mode}} {{submode}} · **Date:** {{date}} · **Hash:** `{{congruence_hash}}`

## Observations
| ID | Timestamp | Observation | Source | Reliability |

## Hypotheses
| ID | Description | Category | Probability | Falsifier |

**Coherence check:** Σ probabilities = 1.00 ✓ [MANDEL-COH]

## ACH Matrix
| | H1 | H2 | H3 | H4 | H5 |
|---|---|---|---|---|---|
| O1 (B) | ++ | + | N | - | -- |
| **Sum** | … | … | … | … | … |
| **Disconfirm count** | 0 | 1 | 2 | 3 | 4 |
| **Weighted (rel.)** | … | … | … | … | … |

**Winner (Heuer disconfirmation rule):** H1 [HEUER-DISCONF]
**Sensitivity:** robust ✓ — no single observation flips the winner
**Most diagnostic:** O3, O7, O1

## Assumptions
| ID | Assumption | Criticality | Testable |

## Falsification
- H1 would be falsified by: …
- H1 would be strengthened by: …

## Limitations
- …

---
*Rules applied: [ICD-203-LC] [ICD-203-SQ] [HEUER-DISCONF] [MANDEL-COH] [NATO-AF]*
```

LIGHT mode trace omits the ACH Matrix block entirely (`ach_matrix: null`).

### 6.3 Tasking view (`assets/templates/tasking_view.md`)

For the follow-up actor.

```markdown
# Tasking — {{analysis_id}}

**Hash:** `{{congruence_hash}}`  (must match analytic trace)

## Collection gaps
- {{gap}}

## Indicators to watch
| Indicator | Threshold | What it changes |

## Peer review requests
- {{request}}

## Conditions that would change the judgment
- If {{if_observed}} → judgment becomes {{judgment_becomes}}
```

### 6.4 LIGHT mode output

LIGHT emits only the decision card with simplifications:

- `confidence` field is omitted (no source quality to anchor it).
- `next_indicator` becomes "what would settle this for me."
- No `[RULE-ID]` markers (LIGHT does not claim doctrinal compliance).
- Footer: `— LIGHT mode · trace available on /sat trace`.

If the user later runs `/sat trace`, the engine *can* emit a thin analytic trace from the same `analysis_id` — not the default. LIGHT respects "bar argument."

### 6.5 Congruence guard

The engine refuses to emit unless:
```
trace.congruence_hash == hash(
    decision_card.bottom_line,
    decision_card.likelihood,
    decision_card.confidence,    # null in LIGHT
    decision_card.top_implication,
)
```

Same check between trace and tasking view. Hash mismatch → engine raises and refuses to render. **The decision card cannot ship with a likelihood that disagrees with the trace.** [ICD-208-CONG] enforced mechanically.

```
assets/templates/
├── decision_card.md         (mandatory both modes)
├── analytic_trace.md        (FULL mandatory, LIGHT optional)
├── analytic_trace_light.md  (LIGHT thin variant)
└── tasking_view.md          (FULL only)
```

## 7 / ACH fixes

Five behavior changes inside `sat_engine/ach.py`.

### 7.1 Heuer disconfirmation rule (winner determination)

**New rule:**
- Compute three values per hypothesis: `sum`, `disconfirm_count`, `weighted`.
- `disconfirm_count` = count of ratings in `{"-", "--"}` for that hypothesis (unweighted).
- `weighted` = Σ (rating_value × reliability_multiplier of observation).
- **Winner = hypothesis with lowest `disconfirm_count`. Ties broken by highest `weighted`.** `sum` is informational only.

The trace surfaces all three so the analyst can see why the winner won.

### 7.2 Reliability weighting (NATO Admiralty A–F → multiplier)

Constants in `sat_engine/ach.py` (tunable, documented):

| NATO Admiralty | Meaning | Multiplier |
|---|---|---|
| A | Completely reliable | 1.0 |
| B | Usually reliable | 0.85 |
| C | Fairly reliable | 0.7 |
| D | Not usually reliable | 0.5 |
| E | Unreliable | 0.3 |
| F | Cannot be judged | 0.1 |

Used by `weighted` score, **not** by `disconfirm_count`. Rationale: a disconfirmation from an F-rated source is still a flag worth seeing — Heuer's principle is conservative on disconfirmation.

### 7.3 Sensitivity analysis (already in script, surface it)

For each observation, recompute scores with that observation removed:
- If winner changes → record `(observation_id, new_winner)` in `flips_on_remove`.
- If no single-observation removal flips the winner → `robust: true`.

The trace's ACH section always shows sensitivity output. Counters the "false precision" failure mode.

### 7.4 Diagnosticity (already in script, surface it)

Per-observation variance across hypothesis ratings. The trace lists the **top 3 most diagnostic observations**. If the top diagnostic is a low-reliability source, that is a visible problem.

(Future improvement noted in `references/doctrine.md`: Heuer's classic measure focuses on inconsistency-count rather than variance. Variance is reasonable; document the alternative.)

### 7.5 Coherence check (new — Mandel 2018, BLOCKING)

```
sum_p = Σ hypothesis.probability for h in hypotheses
if not (0.95 <= sum_p <= 1.05):
    raise CoherenceViolation(sum_p)
```

The engine refuses to proceed past hypothesis generation if probabilities do not sum to ~1.0. Workflow forces the analyst to redistribute. Tagged `[MANDEL-COH]` (BLOCKING).

**Escape hatch:** `sat_engine.ach.allow_overlapping_hypotheses=True` flag for analyses where hypotheses are intentionally non-mutually-exclusive (e.g., "compromised AND insider"). When set, sum constraint relaxes to `[0.95, 2.0]` and a doctrine note `[MANDEL-COH:OVERLAP]` is added.

### 7.6 Vague-hypothesis warning

After scoring, if any hypothesis has:
- ≥3 evidence items rated against it
- All ratings in `{"+", "++"}` (no `-`, `--`, or `N`)

the trace adds a warning:
```
⚠ H2 has all-supporting ratings (5/5). Hypothesis may be too vague — consider splitting or sharpening.
```

### 7.7 Validation summary

| Validation | Layer | Severity |
|---|---|---|
| Schema conformance | `validators.py` | BLOCKING |
| Probability coherence | `ach.py` | BLOCKING (`[MANDEL-COH]`) |
| Reliability set on every observation | `validators.py` | BLOCKING in FULL (`[ICD-203-SQ]`) |
| Likelihood ≠ confidence as fields | schema structural | BLOCKING (`[ICD-203-LC]`) |
| Congruence hash matches | engine emit step | BLOCKING (`[ICD-208-CONG]`) |
| Falsifier set on every hypothesis | schema | BLOCKING |
| Single-source confidence cap | `ach.py` | WARNING |
| Vague hypothesis (all-`+`) | `ach.py` | WARNING |

## 8 / Doctrine catalog & citation enrichment

### 8.1 Catalog file: `doctrine/catalog.v1.json`

Plain JSON, readable by any language. Conforms to `schemas/doctrine.v1.json`.

```json
{
  "schema_version": "1",
  "catalog_version": "2026-05-05.1",
  "rules": [
    {
      "rule_id": "ICD-203-LC",
      "standard": "ICD 203",
      "section": "§c.6",
      "title": "Likelihood and confidence as separate fields",
      "summary": "Analytic products using confidence levels must not combine a confidence level and a degree of likelihood in the same sentence.",
      "citation": "Office of the Director of National Intelligence. ICD 203, Analytic Standards. January 2015. §c.6.",
      "severity_if_violated": "BLOCKING"
    }
    // ... 6 more rules
  ]
}
```

### 8.2 Initial seed (7 rules + 1 variant)

| Rule ID | Standard | Title | Severity |
|---|---|---|---|
| `ICD-203-LC` | ICD 203 §c.6 | Likelihood/confidence as separate fields | BLOCKING |
| `ICD-203-SQ` | ICD 203 §c.4 | Source quality required | BLOCKING (FULL only) |
| `ICD-203-AJ` | ICD 203 §c.7 | Assumptions vs judgments separated | WARNING |
| `ICD-208-CONG` | ICD 208 | Congruence across products | BLOCKING |
| `HEUER-DISCONF` | Heuer 1999 | Rank by fewest disconfirmations | BLOCKING |
| `NATO-AF` | NATO STANAG 2511 | A–F source reliability scale | NOTE |
| `MANDEL-COH` | Mandel 2018 | Probability coherence (Σ ≈ 1.0) | BLOCKING |
| `MANDEL-COH:OVERLAP` | (engine variant) | Overlapping hypotheses allowed | NOTE |

### 8.3 Enrichment script: `sat_engine/enrich.py`

**Inputs:** an analytic trace (markdown or JSON), the catalog.
**Output:** enriched artifact with citations attached.

**Two modes:**

1. **`--mode appendix`** *(default)* — leaves `[RULE-ID]` markers in place, appends `## Citations` at the bottom.
2. **`--mode inline`** — replaces each `[RULE-ID]` with a footnote anchor, emits footnotes at bottom.

Idempotent: running twice produces the same output.

### 8.4 Workflow timing

| Context | When enrichment runs |
|---|---|
| Desktop skill, FULL mode | Optional, on demand via `sat enrich`. Skill suggests it; does not block. |
| Desktop skill, LIGHT mode | Never. LIGHT emits no `[RULE-ID]` markers. |
| Mobile app | At sync time, against the locally-cached catalog. |
| CI / pre-commit | Validation only — every `[RULE-ID]` resolves in catalog. No content modification. |

Write-time analysis never blocks on doctrine lookup. Citations attach later.

### 8.5 Catalog versioning

- File name encodes schema version: `catalog.v1.json`.
- `catalog_version` field is a build identifier (`"2026-05-05.1"`), bumped on any content change.
- Append-only edits (new rules) keep `v1` valid. Old traces still resolve.
- Semantic edits (changing severity or meaning) bump to `catalog.v2.json`. v1 stays alongside.
- Each enriched artifact records the `catalog_version` it was rendered against.

### 8.6 Mobile-app friendliness (concrete)

The catalog is plain JSON conforming to a schema both platforms read:

- iOS (Swift): `JSONDecoder().decode(Catalog.self, from: data)`. No Python.
- Android (Kotlin): `Json.decodeFromString<Catalog>(jsonString)`.
- Both scan markdown for `[A-Z0-9\-:]+` rule_id patterns natively.
- The mobile app's enrichment is the same algorithm in native code, with `tests/parity/` proving it produces identical output to the Python reference.

### 8.7 Future extensions (noted, not built)

- `catalog.v1.{en,fr,de}.json` for translated citation text.
- Industry overlays: `nist-csf.v1.json`, `mitre-attack.v1.json`.
- User-defined rules: `~/.sat/local-catalog.json` for org-specific standards.

## 9 / Testing & parity

Four test layers. Layers 1–3 keep the desktop skill honest. Layer 4 is the mobile-inheritance contract.

### 9.1 Test layout

```
tests/
├── unit/                       Layer 2 — engine logic
│   ├── test_ach.py
```

### 9.2 Layer 1 — schema validation (CI)

- Every `schemas/*.v1.json` validates against the JSON Schema 2020-12 meta-schema.
- Every fixture in `tests/parity/*/expected_*.json` validates against its declared schema.
- Every `[RULE-ID]` referenced in any markdown resolves in `catalog.v1.json`.

### 9.3 Layer 2 — unit tests (regressions for named defects)

`test_ach.py` must include:
- `test_vague_hypothesis_does_not_win` — five hypotheses, H2 has `+` against everything; assert winner is NOT H2.
- `test_disconfirmation_count_breaks_sum_tie` — equal sums, different disconfirm_counts; assert lower disconfirm wins.
- `test_reliability_weights_propagate` — same ratings, change reliability A→F; assert `weighted` changes, `disconfirm_count` does not.
- `test_coherence_violation_raises` — probabilities sum to 0.6; assert `CoherenceViolation` raised.
- `test_sensitivity_detects_flip` — construct case where removing O3 flips winner; assert `sensitivity.flips_on_remove` includes O3.
- `test_overlap_flag_relaxes_coherence` — set `allow_overlapping_hypotheses=True`, sum=1.5; assert no raise.

`test_parsers.py` includes regression: `assert "sqli_attempt" not in entry.tags` (parser emits no interpretation tags).

`test_timeline.py` covers gap detection, rapid succession, configurable thresholds.

`test_validators.py` confirms every BLOCKING rule produces a refusal.

`test_enrich.py` covers appendix vs inline mode, idempotency.

### 9.4 Layer 3 — integration tests (per submode)

Each `test_<submode>_workflow.py` runs the full pipeline and asserts:
- All three artifacts emitted (LIGHT: only decision card).
- `congruence_hash` matches across all three (FULL).
- `[RULE-ID]` markers present in trace (FULL).
- Schema validation passes on every emitted artifact.

### 9.5 Layer 4 — parity tests (mobile contract)

```
tests/parity/breach_001/
├── input/
│   ├── auth.log
│   └── analyst_inputs.json   # hypothesis list, ACH ratings, etc.
├── expected_decision_card.json
├── expected_analytic_trace.json
└── expected_tasking_view.json
```

**Runner contract:** an implementation passes case `breach_001` iff
- given `input/`, it produces three JSON artifacts whose schema-valid fields equal `expected_*.json` (with `analysis_id` and timestamps normalized).
- normalization rules: `analysis_id` → `"<UUID>"`, ISO timestamps → `"<TS>"`, `congruence_hash` recomputed against normalized fields.

**Initial parity set (11 cases):**

| Case | Purpose |
|---|---|
| `breach_001` | BREACH/FULL happy path |
| `crash_001` | CRASH/FULL with timeline gaps |
| `fix_001` | FIX/FULL with diff input |
| `statement_001` | STATEMENT/FULL, no artifacts |
| `general_001` | GENERAL/FULL ambiguous case |
| `light_001` | LIGHT minimal — decision card only |
| `coherence_violation_001` | sum of probs = 0.7 → engine refusal |
| `sensitivity_001` | removing one observation flips winner → flag set |
| `vague_hypothesis_001` | all-`+` hypothesis → warning, not winner |
| `congruence_violation_001` | mismatched fields → engine refusal |
| `reliability_weighting_001` | same ratings, different reliability → different weighted scores |

### 9.6 Test framework

- `pytest` for runner.
- `jsonschema` library for schema validation.
- `deepdiff` (or hand-rolled) for parity comparison with normalization.
- Snapshot testing not used — parity expectations are hand-curated, reviewed when changed.

### 9.7 Acceptance criteria

The redesigned skill ships only when:

1. All schemas validate against meta-schema.
2. All unit tests pass.
3. All 6 integration tests pass.
4. All 11 parity tests produce expected output.
5. Every `[RULE-ID]` in the codebase resolves in the catalog.
6. Orphan-doc check: every reference file is reachable from `SKILL.md` or another reference.
7. "Scripts not bypassed" check: grep for ACH score arithmetic in markdown — should find none in `SKILL.md`.
8. Migration smoke: existing real-world traces (if any) re-run on the new engine produce equivalent or better output (manual review).
